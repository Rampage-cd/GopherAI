# GopherAI
这是一个AI接口调用的项目，现在梳理一下整个项目的架构。

## 1.config包
- 里面只有两个文件，一个go文件，一个toml文件。toml是一种文件格式，相比于json支持注释，相比于yaml不会出现缩进错误（使用的是键值对），非常适合于应用配置文件的场景，同时单独写一个config包也符合配置分离的思想。
- go文件中定义了结构体，使用了toml解析库，每个字段后面的标签，虽然无法被编译器识别，但是可以让解析库按照这个标签将结构体的字段和配置文件中的字段一一对应（将配置文件中的信息加载到结构体字段中）。最后使用了GetConfig函数，确保了不管调用多少次，只执行一次初始化，避免反复读取硬盘文件，极大的提高了性能。

## 2.myjwt包和utils包
- myjwt包定义了jwt的生成与验证函数，前者结合服务器密钥生成token，后者对token进行解析并使用服务器密钥验证token。（token分为三部分，Header.Payload.Signature）
- utils包用于放置一些通用的功能函数，我们这里放置的是，生成给定长度随机数字字符串的函数，MD5函数，UUID函数，以及schema.Message（AI引擎层能够识别的数据格式）和model.Message（自己写的，数据能够识别的数据格式）的相互转化函数。其中MD5是用于保护用户密码的算法，具有单向不可逆性。UUID用于生成永远不会重复的ID，而不需要向数据库去申请。

## 3.model包
- 里面定义了三个文件，分别是管理用户账号信息的user.go，管理聊天界面会话的session.go，以及管理单次消息的message.go。可以看到字段后面的标签不仅有gorm，还有json，前者用于说明该字段存放进入数据库的标准（是否是主键，类型等），后者用于将数据转化为json格式提供给前端显示。这就是模型层（项目的数据基石）。

## 4.common文件夹
- redis包：包含了生成键值对中键的算法（fmt.Sprintf），以及存入验证码键值对，验证验证码的算法。这也是本项目中redis的主要功能（存储并校验验证码）
- mysql包：负责建立与数据库的连接并配置好连接池，最后赋值给外部变量，方便能够被其他包使用。还写了插入用户和查找用户的函数。
- rabbitmq包：负责与rabbitmq服务器建立连接，在连接中创建队列，然后将数据放入队列，以及处理从队列中拿出的数据。还有数据类型转化的函数（json的序列化和反序列化，因为我们代码中使用的是结构体，而消息队列中存放的是json）
- email包：使用了第三方库gomail，帮我们省略了很多底层细节，只需要关注消息的的内容，SMTP服务器的连接（连接的是qq邮箱的专门处理发件业务的服务器），最后就是发送邮件。
- code包：定义了一个新类型Code（本质是int64类型），然后定义了很多Code常量，用于表示状态码，再定义了一个哈希表，将常量与对应的错误信息对应。后面绑定了两个方法，分别用于获取状态码和状态码对应的错误信息。
- image包；定义了图像识别的结构体以及处理逻辑，关键部分在于结构体的初始化以及如何处理图像，需要用到图像处理方面的知识，另外需要使用onnx文件来初始化推理会话。
- aihelper包：

    1. model.go文件使用了字节跳动CloudWeGo团队的eino框架，一个专门用于构建AI应用的框架，好处是对于不同厂商的模型接口，不需要各自写一套JSON结构，只用统一转化成schema.Message这个标准格式，框架会帮你转化成不同厂商各自需要的格式，还有就是对于流式输出中需要建立复杂长链接的逻辑，被集成在了Stream()函数中，不断调用Recv()函数即可获取数据。模型调用使用的是openai（云端）和ollama（本地）。
    2. aihelper.go文件：定义AIHelper数据结构，负责保存消息，消息的持久化，以及调用AI模型回复用户的问题（分为一次性输出和流式输出两种输出方式）
    3. factory.go文件：定义AIModelFactory数据结构，负责保存不同模型的创建方法，以及根据方法创建模型，根据创建模型的方法创建AIHelper，还支持模型创建方法的扩展。
    4. manager.go文件：定义AIHelperManager数据结构，负责管理不同用户的不同会话中用到的AIHelper结构体，并绑定了一些查找和删除方法。（3与4都使用了全局的实例和单次初始化的逻辑）

    总结一下，model是最底层的AI引擎创建，aihelper则负责用户的一次会话，里面包含了AI引擎，因为要通过它来给用户生成回复，factory负责根据不同的模型执行对应的创建函数，避免冗长的if...else语句，还方便代码的维护和扩展，而manager则是全局管理，管理所有用户的所有会话，如果没有创建会话，还会使用factory中的函数创建。

## 5.dao文件夹
- 里面的三个文件，user.go，session.go，message.go组成了项目的DAO层（数据访问对象层），负责把底层的数据库操作封装成简单的函数，供上层业务调用。
user.go负责处理账号相关的数据库操作，session.go负责处理对话相关的数据库操作，message.go负责处理与一条聊天相关的数据库操作。

## 6.service文件夹
- 里面的三个文件，image.go，session.go，user.go组成成了项目的Service层，负责把底层的功能函数拼接成一个完整的功能，并对外提供服务。
image.go负责针对传入的文件头，创建图像识别器并调用底层的图像识别方法识别图片，user.go负责处理用户的注册，登录，接收验证码等操作，session.go负责创建会话，生成回答，查找会话，其中回答方式又分为流式输出和普通输出。

## 7.controller文件夹
- 首先common.go文件定义了一个响应的结构体，用于统一返回给前端的JSON格式。其次image.go，session.go，user.go是对上一层（service层）函数的封装，用于解析HTTP报文，并将解析中的参数传入上一层的函数中，实现了不同模块的解耦。之所以在每个文件中定义了许多结构体，是因为要控制输入和输出，同时也是我们参数（解析请求后的）和响应（处理完请求后的）所存放的地方。

## 8.middleware/jwt文件
- jwt包：作为中间件负责拦截HTTP请求，并校验请求头中的jwt（调用了底层myjwt包中的方法），校验成功则放行，否则直接中断请求。

## 9.router包
- 整个后端的最外层——路由层。负责实现路由分组，绑定中间件，以及为不同的路径绑定不同的处理函数。请求只有两种类型GET和POST，两者的区别在于参数的位置不同，一个位于请求头中的url中，一个位于请求体中，以及产生的结果不同，前者不会更改数据库，后者则会。
    
    那么为什么Login和ChatHistory并不改变数据库，但是却要用POST呢？因为Login的参数属于敏感信息，而HTTPS只能保护好请求体，不在乎请求头，所以要使用POST将该信息放到请求体中，传给后端；而ChatHistory中由于筛选逻辑的复杂性，最好也是放在请求体中传给后端，避免url变得臃肿。

## 10.main.go
- 是编译的唯一入口，通过`go build main.go`，编译器会顺藤摸瓜的把相关的所有模块打包成一个二进制文件，同时按顺序启动了各种模块（加载配置->连接MySQL，Redis，RabbitMQ->注册路由->启动服务），他把我们写的所有模块连接了起来，组成了一个完整的后端服务。


## 项目运行
- 需要注意，虽然模型用的是openai的模型，也就是说数据格式是按照openai的标准，但是很多厂商都兼容openai的标准，所以可以直接把OPENAI_API_KEY等三个参数填写成阿里的，使用免费的模型。

## 我的发现
- 在aihelper.go文件中，发现在初始化函数中，调用rabbitmq包中结构体时，可能存在对应结构体没有初始化的问题，调用绑定函数前加上了一个判断来防止panic。在AddMessage函数中，修改messages没有加锁，补充上了。最后也是最重要的优化部分，在让AI生成回答前，只选取最近的20条对话记录作为输入，很好的防止了随着对话次数增加输入过大的问题。

- 在main.go文件中，发现redis和rabbitmq基础组件的错误处理逻辑没有书写，重新完善了common文件夹中这两个组件的函数逻辑，并将错误处理逻辑同步到main.go文件中。
